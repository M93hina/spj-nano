# B1F混雑度推定・予測システム 作業計画

事前分析(`docs/B1F事前分析レポート.md`)の結論を受けた、プロトタイプ化までの作業計画。
既存構成(`spj_nano/` + `scripts/` + `app.py` Streamlitダッシュボード)を拡張する形で進める。

## フェーズ1: 前処理モジュールの実装  ✅ 完了(2026-07-08)

分析で判明したデータ品質の問題に対処する層を作る。以降のすべての処理はこの前処理済みデータを使う。

- [x] `spj_nano/preprocess.py` を新規作成
  - [x] **スパイク除去**: 変化率フィルタ(例: 5分間で±300ppm超の単発〜2点スパイクを除去し補間)。6/29の2797ppm・2443ppmが除去されることをテストケースにする
  - [x] **無人時ベースライン算出**: 直近N日の休日・早朝データからB1Fベースライン(約460〜470ppm)をローリング推定する関数。深夜帯を使う場合は+30ppm補正を織り込む
  - [x] **センサーオフセット補正**: 深夜1-5時の中央値を用いた各センサーのオフセット推定(B1F=497, 1F=473, 3F=464, 4F=465 等)。他階を参照信号に使う際の正規化に利用
- [x] 前処理の動作確認スクリプト(処理前後の比較を出力) → `scripts/check_preprocess.py`

**実装メモ**: スパイクは「急激なジャンプ+max_width点以内の復帰」で検出(減衰尾3-4点も補間)。6/29は11:55-12:10(4点)・15:55-16:05(3点)=計7点を除去、処理後max=901ppm(事前分析と一致)。ベースライン中央値470ppm、オフセットは事前分析値と完全一致。

## フェーズ2: 混雑レベル閾値の再キャリブレーション  ✅ 完了(2026-07-08)

- [x] `app.py` の固定閾値(500/700/1000)をデータ駆動閾値 **550/650/760ppm** の4段階に差し替え
  - レベル1: 〜550ppm「空いています」/ レベル2: 550〜650「やや利用あり」/ レベル3: 650〜760「混雑」/ レベル4: 760〜「非常に混雑」
- [x] 閾値をハードコードせず、設定 or 分布からの自動算出(分位点ベース)として実装し、データ蓄積に応じて更新可能にする → `spj_nano/levels.py`(平日日中9-18時の Q30/Q55/Q90)
- [x] 過去1ヶ月分に新旧閾値を適用したときのレベル分布の比較を確認(新閾値で4段階が実際に出現することを検証) → `scripts/check_levels.py`

**実装メモ**: データ駆動(自動)=526/633/748ppmで提案手動値に近い。平日日中で旧閾値はL4が0%出現(事前分析と一致)、新閾値(手動/自動)は4段階すべて出現。

## フェーズ3: 混雑予測モデル(ベースライン版)  ✅ 完了(2026-07-08)

分析より、曜日×時刻の過去平均だけでR²相当0.845。まずこれをベースラインとして実装する。

- [x] `spj_nano/forecast.py` を新規作成
  - [x] **曜日×時刻ベースライン**: 曜日×5分(または15分)ビンの過去平均(直近4〜6週の加重平均)を予測値とする → `BaselineProfile`(half_life_weeks=2.0で直近週重み付け)
  - [x] **当日残差補正**: 「現在の実測値 − ベースライン」の残差を先の予測に減衰させながら乗せるハイブリッド化(直近の混み具合を反映) → `forecast_horizon`(tau_hours=2.0)
  - [x] 学期進行による上昇トレンド(水曜: 6/3→6/24で約+100ppm)への対応: 直近週に重みを置く、またはトレンド項を追加 → 加重平均のhalf_lifeで対応
- [x] **評価**: 直近1週間をホールドアウトし、ベースライン単体 vs ハイブリッドのMAE/残差stdを比較 → `scripts/check_forecast.py`
- [x] 予測の出力: 「+1h / +2h / +3h後の予測CO2とレベル」を返すAPIをモジュールとして整備 → `forecast_horizon()`

**評価結果**: ベースライン単体 MAE=41.7/std=49.4(R²相当0.770)。ハイブリッドは +1h: MAE16.5/std19.5、+2h: MAE26.6/std31.4、+3h: MAE32.8/std38.7。**目標(残差std≤45ppm)を全horizonで達成**。

### バグ修正(2026-07-08 追記)

- **不具合**: `spj_nano/levels.py` の `Thresholds.level()` が、CO2がL4域(閾値t3以上、実データの4.64%)のとき `IndexError` でクラッシュしていた。`bounds` プロパティが `(-inf, t1, t2, t3)` の4要素しかないのに、ループ最終周で `bounds[4]` を参照していたのが原因。前述の「AppTestで例外なしを検証済み」は、検証時にたまたまL4域の値を通らなかったため見逃していた(境界値・全レベル網羅のテストではなかった)。
- **修正**: `bounds` を `(-inf, t1, t2, t3, +inf)` の5要素にし、`classify_series`(`np.digitize`実装)と境界挙動(閾値ちょうどは上位レベル)を一致させた。あわせて `Thresholds.level(800)` 相当のL4域を通す単体テスト・AppTest経路を追加して再発防止。
- **あわせて対応**: `app.py` にデータ鮮度警告(最終データが30分以上古い場合に `st.warning` 表示、予測見出し・キャプションにも最終データ時点である旨を明記)を追加。未使用の `_level_color()` を削除。
